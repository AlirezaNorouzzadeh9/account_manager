# DO Account Manager Bot

ربات تلگرامی (Laravel + [nutgram/laravel](https://github.com/nutgram/laravel)) برای مدیریت اکانت‌های DigitalOcean: ساخت سرور، روشن/خاموش/ریبوت، ریسایز، ریبیلد، آی‌پی اضافه و تغییر آی‌پی.

معماری طوری طراحی شده که افزودن دیتاسنترهای دیگر (Vultr، Linode، ...) در آینده فقط نیاز به یک کلاینت جدید دارد، بدون تغییر در فلوی ربات.

## پیش‌نیاز

- PHP 8.3+‎ و Composer
- MySQL
- برای اجرای واقعی: یک دامنه با SSL معتبر (Telegram webhook باید HTTPS باشد). برای تست لوکال می‌توانید از ngrok استفاده کنید.

## نصب

```bash
composer install
cp .env.example .env
php artisan key:generate
```

مقادیر زیر را در `.env` تنظیم کنید:

```
DB_DATABASE=account_manager
DB_USERNAME=root
DB_PASSWORD=

TELEGRAM_TOKEN=<توکن ربات از @BotFather>
ADMIN_TELEGRAM_IDS=<آیدی عددی تلگرام شما، با کاما جدا برای چند نفر>
```

سپس دیتابیس را بسازید و مایگریشن را اجرا کنید:

```bash
mysql -u root -e "CREATE DATABASE account_manager CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
php artisan migrate
```

## اجرا

دو پردازش باید همزمان در حال اجرا باشند:

```bash
# پردازش صف؛ برای پیگیری وضعیت عملیات‌های async روی DigitalOcean (ساخت/ریسایز/ریبیلد/...) ضروری است
php artisan queue:work

# وب‌سرور (یا از nginx/apache روی دامنه‌ی خودتان استفاده کنید)
php artisan serve
```

سپس وبهوک تلگرام را روی آدرس عمومی پروژه ثبت کنید:

```bash
php artisan nutgram:hook:set https://yourdomain.com/api/telegram/webhook
```

برای توسعه/تست لوکال بدون دامنه، می‌توانید به‌جای وبهوک از حالت polling استفاده کنید:

```bash
php artisan nutgram:run
```

## استفاده از ربات

- `/start` — نمایش منوی اصلی (پنل‌های من / ساخت سرور / سرورهای من)
- `/cancel` — لغو هر عملیات نیمه‌کاره

هنگام افزودن پنل، ابتدا دیتاسنتر (فعلاً فقط DigitalOcean فعال است) و سپس توکن API را از شما می‌پرسد. توکن را می‌توانید از اینجا بسازید:
https://cloud.digitalocean.com/account/api/tokens (دسترسی Read & Write لازم است)

توکن پس از اعتبارسنجی به‌صورت رمزنگاری‌شده (`encrypted` cast روی مدل `Panel`) در دیتابیس ذخیره می‌شود و پیام حاوی توکن از تاریخچه چت پاک می‌شود.

## نود کردن سرور به PasarGuard

از داخل «اطلاعات سرور» دکمه‌ی «🧩 نود کردن به پاسارگارد» یک نود PasarGuard (`pasarguard/node`) روی همان سرور بالا می‌آورد: در صورت نبود Docker نصبش می‌کند، فایل‌های `docker-compose.yml`/`node-1/.env` را می‌سازد، یک گواهی SSL مخصوص همان سرور تولید می‌کند و `docker compose up -d` می‌زند. اگر کانفیگ وایرگاردی در تنظیمات ذخیره کرده باشید، همان‌ها هم روی سرور فعال می‌شوند.

اگر رمز روت سرور ذخیره نشده باشد (سرورهایی که قبل از این قابلیت ساخته شده‌اند) یا رمز ذخیره‌شده دیگر کار نکند، ربات همان لحظه رمز فعلی روت را از شما می‌پرسد و ذخیره‌اش می‌کند.

`API_KEY` نود برای همه‌ی نودهایی که این ربات می‌سازد ثابت است (در `.env` تنظیم می‌شود، هرگز در کد/گیت قرار نمی‌گیرد):

```
PASARGUARD_API_KEY=<uuid ثابت>
```

گواهی TLS نود **ثابت نیست** — هر نود گواهی self-signed مخصوص به خودش را (با IP واقعی همان سرور در subjectAltName) موقع نصب تولید می‌کند، و همیشه هم با همین IP (نه دامنه) در پنل PasarGuard ثبت می‌شود، چون پنل گواهی نود را در برابر آدرسی که باهاش وصل می‌شود اعتبارسنجی می‌کند و یک گواهی/آدرس ناهم‌خوان خطای `certificate verify failed` می‌دهد.

### گواهی ثابت با DNS (فعلاً غیرفعال)

زیرساخت یک حالت جایگزین هم تو کد هست (`CloudflareDnsClient`، `NodeCertificate`، تنظیمات `config/dns.php`) که در صورت فعال بودن، به‌جای گواهی مخصوص هر IP، یک ساب‌دامین Cloudflare به اسم پروفایل وایرگارد (مثلاً `germany.node.pcbot.top`) بسازد و یک گواهی wildcard مشترک بین همه‌ی نودها استفاده کند — هدفش این بود که دامنه (نه IP) در پنل ثبت شود تا با «🔄 تغییر سرور» عوض نشود. این حالت **فعلاً هیچ‌جا فراخوانی نمی‌شود** (نصب نود همیشه با IP کار می‌کند)، پس مقداردهی `CLOUDFLARE_API_TOKEN`/`CLOUDFLARE_ZONE_ID` در `.env` هیچ اثری ندارد. اگر بعداً لازم شد، کافی‌ست در `InstallPasarguardNodeJob`/`ReplaceServerFinishJob` اسم پروفایل هم به `PasarguardNodeInstaller::install()` پاس داده شود.

## تنظیمات و وایرگارد

از `/start` → «⚙️ تنظیمات» → «🔒 وایرگاردها» دو چیز جدا مدیریت می‌شود:

- **لوکیشن‌ها**: لیست تخت و global. هر لوکیشن فقط شامل چیزی است که واقعاً بین لوکیشن‌ها فرق می‌کند — **اسم، کد دو حرفی کشور (اختیاری)، آی‌پی، PublicKey سرور** (با «➕ افزودن لوکیشن»). کد کشور (مثلاً `DE`) فقط برای نمایش پرچم است — از «🌍 تنظیم کشور» هم قابل تغییر است. `Address`/`DNS`/`AllowedIPs`/پورت `Endpoint`/`Table` بین همه‌ی لوکیشن‌ها ثابت است (مقداردهی‌شده در کد، در `PasarguardNodeInstaller`) و جایی برای تنظیم از داخل ربات ندارد.
- **پروفایل‌ها** («🪪 پروفایل‌ها»): هویت هر سرور — فقط **اسم + PrivateKey**. موقع «🧩 نود کردن به پاسارگارد» یا «🔄 بروزرسانی وایرگاردها» روی هر سروری، اول می‌پرسد کدام پروفایل (یا «🚫 بدون وایرگارد») روی همان سرور فعال شود؛ همان PrivateKey برای تمام لوکیشن‌های آن سرور استفاده می‌شود (یک فایل/اینترفیس جدا برای هر لوکیشن، مثلاً `germany` → `/etc/wireguard/germany.conf`). این یعنی سرورهای مختلف می‌توانند هویت‌های متفاوتی داشته باشند، ولی همه به همان مجموعه‌ی لوکیشن‌ها وصل می‌شوند.

## چک پینگ ایران بعد از ساخت سرور

به محض اینکه یک سرور جدید ساخته و آماده شد، ربات خودکار از طریق API عمومی [check-host.net](https://check-host.net) از نودهای ایرانی (اصفهان/شیراز×۲/تهران×۳/قم/خنج) پینگ می‌گیرد.

اگر پینگ کامل تمیز نبود (حداقل یک نود جواب نداد)، به‌جای گزارش همون سرور، ربات خودکار سرور را حذف و یکی تازه با همان مشخصات می‌سازد و دوباره چک می‌کند — تا سقف ۱۰ تلاش. در این حلقه همیشه **حداکثر ۲ سرور** همزمان وجود دارد: «بهترین سرور پیدا‌شده تا الان» + سروری که تازه ساخته شده و دارد چک می‌شود؛ هرکدام امتیاز (تعداد نودهای ایرانی که پینگ موفق داشتند) کمتری داشت حذف می‌شود و آن یکی به‌عنوان بهترین باقی می‌ماند. به محض رسیدن به یک پینگ کاملاً تمیز، یا تمام‌شدن ۱۰ تلاش (که در این صورت بهترین سروری که تا آن لحظه پیدا شده گزارش می‌شود، حتی اگر ۱۰۰٪ تمیز نباشد)، پیام «سرور آماده شد» با نتیجه‌ی پینگ فرستاده می‌شود.

پیام «سرور آماده شد» یک دکمه‌ی «🔄 تغییر سرور» هم دارد **فقط وقتی که پینگ نهایی کامل تمیز نبود** (اگر تمیز بود، همان حلقه‌ی خودکار بالا قبلاً کارش را کرده و دکمه‌ی دستی اضافه‌ای لازم نیست). این سرور تازه ساخته شده و چیزی رویش نصب نشده، پس زدن این دکمه بعد از تایید فقط همان سرور را حذف و با همان مشخصات دوباره می‌سازد (بدون نود/وایرگارد، بدون حلقه‌ی خودکار — یک تلاش ساده) — نتیجه هم دقیقاً مثل ساخت اول گزارش می‌شود.

این با دکمه‌ی «🔄 تغییر سرور» که چک دوره‌ای پینگ (پایین‌تر) یا صفحه‌ی «اطلاعات سرور» برای یک سرور از قبل در حال استفاده نشان می‌دهد فرق دارد: آنجا چون احتمالاً نود/وایرگارد واقعی روی سرور نصب است، ابتدا یک سرور **جدید** ساخته می‌شود (سرور فعلی دست‌نخورده می‌ماند) و همان حلقه‌ی «بهترین از بین ۲ سرور، تا سقف ۱۰ تلاش» بالا روی آن اجرا می‌شود. به محض پیدا شدن نتیجه (تمیز یا بهترینِ ۱۰ تلاش)، نود پاسارگارد + همان پروفایل وایرگارد سرور قبلی روی سرور جدید پیاده می‌شود؛ اگر موفق بود، سرور قبلی خودکار تا ۵ دقیقه‌ی دیگر حذف می‌شود (فرصتی برای بررسی/لغو)، و اگر ناموفق بود، سرور قبلی دست‌نخورده می‌ماند و با دکمه‌ی جدای «🗑 بله، سرور قبلی حذف شود» تاییدِ حذف از شما گرفته می‌شود.

### چک دوره‌ای پینگ سرورها

دستور `php artisan servers:check-pings` (هر ۱۰ دقیقه، از طریق crontab سرور) پینگ همه‌ی سرورهایی که این ربات مشخصاتشان را ذخیره کرده بررسی می‌کند. اگر پینگ مشکلی نداشت هیچ پیامی فرستاده نمی‌شود؛ اگر مشکل داشت، همان هشدار + دکمه‌ی «🔄 تغییر سرور» برای همه‌ی ادمین‌ها ارسال می‌شود تا خودشان تصمیم بگیرند (با زدنش، همان فلوی «تغییر سرور» بالا اجرا می‌شود).

## افزودن یک دیتاسنتر جدید (مثلاً Vultr یا Linode)

1. یک کلاینت جدید در `app/Services/Providers/<Name>/<Name>Client.php` بسازید که `App\Services\Providers\ProviderClient` را پیاده‌سازی کند (متدهای regions/sizes/images/createServer/... مطابق API آن سرویس).
2. در `app/Enums/Provider.php` متد `isAvailable()` را برای آن مورد `true` کنید.
3. کلاینت جدید را به نگاشت `ProviderManager::$clients` در `app/Services/Providers/ProviderManager.php` اضافه کنید.

فلوی ربات (افزودن پنل، ساخت سرور، مدیریت سرور) بدون هیچ تغییری با دیتاسنتر جدید کار خواهد کرد.

## تست

```bash
php artisan test
```

تست‌ها فلوی افزودن پنل، ساخت سرور و اکشن‌های سرور را با موک کردن API دیجیتال اوشن (`Http::fake`) و ربات تلگرام (`Nutgram::fake`) پوشش می‌دهند. تست‌ها روی یک دیتابیس MySQL جدا (`account_manager_test`) اجرا می‌شوند؛ در صورت نیاز آن را بسازید:

```bash
mysql -u root -e "CREATE DATABASE account_manager_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```
