# DO Account Manager Bot

ربات تلگرامی (Laravel + [nutgram/laravel](https://github.com/nutgram/laravel)) برای مدیریت اکانت‌های DigitalOcean: ساخت سرور، روشن/خاموش/ریبوت، ریسایز، ریبیلد، آی‌پی اضافه و تغییر آی‌پی.

معماری طوری طراحی شده که افزودن دیتاسنترهای دیگر (Vultr، Linode، ...) در آینده فقط نیاز به یک کلاینت جدید دارد، بدون تغییر در فلوی ربات.

## پیش‌نیاز

- PHP 8.3+‎ و Composer
- MySQL
- برای اجرای واقعی: یک دامنه با SSL معتبر (Telegram webhook باید HTTPS باشد). برای تست لوکال می‌توانید از ngrok استفاده کنید.

## نصب

```bash
composer install
cp .env.example .env
php artisan key:generate
```

مقادیر زیر را در `.env` تنظیم کنید:

```
DB_DATABASE=account_manager
DB_USERNAME=root
DB_PASSWORD=

TELEGRAM_TOKEN=<توکن ربات از @BotFather>
ADMIN_TELEGRAM_IDS=<آیدی عددی تلگرام شما، با کاما جدا برای چند نفر>
```

سپس دیتابیس را بسازید و مایگریشن را اجرا کنید:

```bash
mysql -u root -e "CREATE DATABASE account_manager CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
php artisan migrate
```

## اجرا

دو پردازش باید همزمان در حال اجرا باشند:

```bash
# پردازش صف؛ برای پیگیری وضعیت عملیات‌های async روی DigitalOcean (ساخت/ریسایز/ریبیلد/...) ضروری است
php artisan queue:work

# وب‌سرور (یا از nginx/apache روی دامنه‌ی خودتان استفاده کنید)
php artisan serve
```

سپس وبهوک تلگرام را روی آدرس عمومی پروژه ثبت کنید:

```bash
php artisan nutgram:hook:set https://yourdomain.com/api/telegram/webhook
```

برای توسعه/تست لوکال بدون دامنه، می‌توانید به‌جای وبهوک از حالت polling استفاده کنید:

```bash
php artisan nutgram:run
```

## استفاده از ربات

- `/start` — نمایش منوی اصلی (پنل‌های من / ساخت سرور / سرورهای من)
- `/cancel` — لغو هر عملیات نیمه‌کاره

هنگام افزودن پنل، ابتدا دیتاسنتر (فعلاً فقط DigitalOcean فعال است) و سپس توکن API را از شما می‌پرسد. توکن را می‌توانید از اینجا بسازید:
https://cloud.digitalocean.com/account/api/tokens (دسترسی Read & Write لازم است)

توکن پس از اعتبارسنجی به‌صورت رمزنگاری‌شده (`encrypted` cast روی مدل `Panel`) در دیتابیس ذخیره می‌شود و پیام حاوی توکن از تاریخچه چت پاک می‌شود.

## نود کردن سرور به PasarGuard

از داخل «اطلاعات سرور» دکمه‌ی «🧩 نود کردن به پاسارگارد» یک نود PasarGuard (`pasarguard/node`) روی همان سرور بالا می‌آورد: در صورت نبود Docker نصبش می‌کند، فایل‌های `docker-compose.yml`/`node-1/.env` را می‌سازد، یک گواهی SSL مخصوص همان سرور تولید می‌کند و `docker compose up -d` می‌زند. اگر کانفیگ وایرگاردی در تنظیمات ذخیره کرده باشید، همان‌ها هم روی سرور فعال می‌شوند.

اگر رمز روت سرور ذخیره نشده باشد (سرورهایی که قبل از این قابلیت ساخته شده‌اند) یا رمز ذخیره‌شده دیگر کار نکند، ربات همان لحظه رمز فعلی روت را از شما می‌پرسد و ذخیره‌اش می‌کند.

`API_KEY` نود برای همه‌ی نودهایی که این ربات می‌سازد ثابت است (در `.env` تنظیم می‌شود، هرگز در کد/گیت قرار نمی‌گیرد):

```
PASARGUARD_API_KEY=<uuid ثابت>
```

گواهی TLS نود به‌طور پیش‌فرض **ثابت نیست** — هر نود گواهی self-signed مخصوص به خودش را (با IP واقعی همان سرور در subjectAltName) موقع نصب تولید می‌کند، چون پنل گواهی نود را در برابر آدرسی که باهاش وصل می‌شود اعتبارسنجی می‌کند و یک گواهی مشترک با IP اشتباه خطای `certificate verify failed` می‌دهد.

### گواهی ثابت (اختیاری) با DNS

اگر می‌خواهید گواهی نودها ثابت بماند، یک دامنه‌ی مدیریت‌شده در Cloudflare لازم است. این سه مقدار را در `.env` تنظیم کنید:

```
CLOUDFLARE_API_TOKEN=<توکن با دسترسی Zone → DNS → Edit روی همان zone>
CLOUDFLARE_ZONE_ID=<Zone ID دامنه‌ی موردنظر>
CLOUDFLARE_NODE_DOMAIN=node.pcbot.top
```

توکن API کلادفلر باید دقیقاً پرمیشن **«DNS» با سطح Edit** (زیر دسته‌ی «DNS & Zones» موقع ساخت توکن) روی همان zone را داشته باشد — پرمیشن‌های مشابه مثل «DNS View»، «DNS Firewall» یا «Zone Settings» کافی نیستند و با خطای `Authentication error` رد می‌شوند. `CLOUDFLARE_ZONE_ID` هم باید Zone ID باشد، نه Account ID (فرمت هر دو یک رشته‌ی ۳۲کاراکتری هگزادسیمال است، پس به‌راحتی جابه‌جا می‌شوند).

این قابلیت فقط وقتی فعال می‌شود که **یک پروفایل وایرگارد** هم به همان سرور اختصاص داده شده باشد (چون اسم پروفایل، اسم زیردامنه هم می‌شود؛ سروری که پروفایلی ندارد، هویت پایداری برای دامنه‌دار شدن ندارد و همیشه گواهی مخصوص IP خودش را می‌گیرد). به همین دلیل هم اسم پروفایل موقع ساخت فقط حروف انگلیسی/عدد/خط‌تیره را قبول می‌کند.

موقع نود کردن سروری با پروفایل انتخاب‌شده: ربات خودکار یک رکورد A به‌شکل `<اسم پروفایل>.<CLOUDFLARE_NODE_DOMAIN>` می‌سازد که به IP همان سرور اشاره می‌کند (مثلاً پروفایل `germany` → `germany.node.pcbot.top`)، و به‌جای گواهی مخصوص IP، یک گواهی self-signed wildcard (`*.<CLOUDFLARE_NODE_DOMAIN>`) تولید می‌کند که فقط **یک‌بار** ساخته و برای همه‌ی نودها استفاده می‌شود. چون دامنه به اسم پروفایل وصل است نه به یک سرور خاص، اگر بعداً همان سرور با «🔄 تغییر سرور» جایگزین شود، دامنه بدون کار اضافه دنبال همان پروفایل روی سرور جدید می‌رود. در پیام نتیجه‌ی نصب، آدرسی که باید به‌جای IP در پنل PasarGuard ثبت کنید نشان داده می‌شود. اگر ثبت DNS به هر دلیلی (توکن/دسترسی اشتباه) ناموفق باشد، دلیلش در همان پیام نشان داده می‌شود و گواهی مخصوص IP جایگزین می‌شود. اگر این سه مقدار خالی بمانند یا سروری پروفایلی نداشته باشد، رفتار قبلی (گواهی جدا برای هر IP) بدون تغییر باقی می‌ماند.

## تنظیمات و وایرگارد

از `/start` → «⚙️ تنظیمات» → «🔒 وایرگاردها» دو چیز جدا مدیریت می‌شود:

- **لوکیشن‌ها**: لیست تخت و global. هر لوکیشن فقط شامل چیزی است که واقعاً بین لوکیشن‌ها فرق می‌کند — **اسم، آی‌پی، PublicKey سرور** (با «➕ افزودن لوکیشن»). `Address`/`DNS`/`AllowedIPs`/پورت `Endpoint`/`Table` بین همه‌ی لوکیشن‌ها ثابت است (مقداردهی‌شده در کد، در `PasarguardNodeInstaller`) و جایی برای تنظیم از داخل ربات ندارد.
- **پروفایل‌ها** («🪪 پروفایل‌ها»): هویت هر سرور — فقط **اسم + PrivateKey**. موقع «🧩 نود کردن به پاسارگارد» یا «🔄 بروزرسانی وایرگاردها» روی هر سروری، اول می‌پرسد کدام پروفایل (یا «🚫 بدون وایرگارد») روی همان سرور فعال شود؛ همان PrivateKey برای تمام لوکیشن‌های آن سرور استفاده می‌شود (یک فایل/اینترفیس جدا برای هر لوکیشن، مثلاً `germany` → `/etc/wireguard/germany.conf`). این یعنی سرورهای مختلف می‌توانند هویت‌های متفاوتی داشته باشند، ولی همه به همان مجموعه‌ی لوکیشن‌ها وصل می‌شوند.

## چک پینگ ایران بعد از ساخت سرور

به محض اینکه یک سرور جدید ساخته و آماده شد، ربات خودکار از طریق API عمومی [check-host.net](https://check-host.net) از ۵ نود ایرانی (اصفهان/تهران×۳/خنج) پینگ می‌گیرد و نتیجه (میانگین تأخیر هر نود، یا «بدون پاسخ») را همراه پیام «سرور آماده شد» می‌فرستد. این کاملاً best-effort است — اگر چک هاست در دسترس نباشد، ساخت سرور را ناموفق نمی‌کند.

پیام «سرور آماده شد» همیشه یک دکمه‌ی «🔄 تغییر سرور» هم دارد (چه پینگ کامل باشد چه نباشد) — چون آی‌پی یک سرور بعد از ساخت قابل تغییر نیست، تنها راه گرفتن آی‌پی جدید حذف و ساخت دوباره است. این سرور تازه ساخته شده و چیزی رویش نصب نشده، پس زدن این دکمه بعد از تایید فقط همان سرور را حذف و با همان مشخصات دوباره می‌سازد (بدون نود/وایرگارد) — نتیجه هم دقیقاً مثل ساخت اول، با همین دکمه‌ی «🔄 تغییر سرور» گزارش می‌شود.

این با دکمه‌ی مشابهی که چک دوره‌ای پینگ (پایین‌تر) برای یک سرور از قبل در حال استفاده می‌فرستد فرق دارد: آنجا چون احتمالاً نود/وایرگارد واقعی روی سرور نصب است، ابتدا یک سرور **جدید** ساخته می‌شود (سرور فعلی دست‌نخورده می‌ماند)، اگر پینگ سرور جدید هم مشکل داشت خودکار حذف و دوباره ساخته می‌شود (حداکثر ۳ تلاش)، و به‌محض پینگ سالم، نود پاسارگارد + همان پروفایل وایرگارد سرور قبلی روی سرور جدید پیاده می‌شود؛ فقط در آخر، با دکمه‌ی جدای «🗑 بله، سرور قبلی حذف شود»، سرور قبلی حذف می‌شود.

### چک دوره‌ای پینگ سرورها

دستور `php artisan servers:check-pings` (هر ۳۰ دقیقه، از طریق crontab سرور) پینگ همه‌ی سرورهایی که این ربات مشخصاتشان را ذخیره کرده بررسی می‌کند. اگر پینگ مشکلی نداشت هیچ پیامی فرستاده نمی‌شود؛ اگر مشکل داشت، همان هشدار + دکمه‌ی «🔄 تغییر سرور» برای همه‌ی ادمین‌ها ارسال می‌شود تا خودشان تصمیم بگیرند.

## افزودن یک دیتاسنتر جدید (مثلاً Vultr یا Linode)

1. یک کلاینت جدید در `app/Services/Providers/<Name>/<Name>Client.php` بسازید که `App\Services\Providers\ProviderClient` را پیاده‌سازی کند (متدهای regions/sizes/images/createServer/... مطابق API آن سرویس).
2. در `app/Enums/Provider.php` متد `isAvailable()` را برای آن مورد `true` کنید.
3. کلاینت جدید را به نگاشت `ProviderManager::$clients` در `app/Services/Providers/ProviderManager.php` اضافه کنید.

فلوی ربات (افزودن پنل، ساخت سرور، مدیریت سرور) بدون هیچ تغییری با دیتاسنتر جدید کار خواهد کرد.

## تست

```bash
php artisan test
```

تست‌ها فلوی افزودن پنل، ساخت سرور و اکشن‌های سرور را با موک کردن API دیجیتال اوشن (`Http::fake`) و ربات تلگرام (`Nutgram::fake`) پوشش می‌دهند. تست‌ها روی یک دیتابیس MySQL جدا (`account_manager_test`) اجرا می‌شوند؛ در صورت نیاز آن را بسازید:

```bash
mysql -u root -e "CREATE DATABASE account_manager_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```
